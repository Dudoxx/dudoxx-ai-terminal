### 0.B вҖ” CANNOT TYPE: per-char input never submits (gateway Enter handling)

**Status: ROOT-CAUSED + REPRODUCED, not yet fixed.**

### Reproduction (deterministic)
```sh
# broker 6481 + web 3460 must be up (Node 22)
TID=$(curl -s -X POST http://localhost:6481/api/v1/terminals -H 'content-type: application/json' -d '{"title":"t"}' | jq -r .terminalId)
# Simulate the BROWSER input path: per-char, raw \r, NO enter flag
node -e 'const W=require("./ddx-term-web/node_modules/ws");const ws=new W("ws://localhost:3460/term/"+process.argv[1]);
ws.on("open",()=>setTimeout(()=>{for(const c of ["l","s","\r"])ws.send(JSON.stringify({type:"input",terminalId:process.argv[1],data:c}))},400));
ws.on("message",m=>{if(/total |DEMO/.test(m.toString()))console.log("ECHO")});
setTimeout(()=>process.exit(0),4000);' "$TID"
# вҶ’ NO echo. Command never executes.
```
Contrast: `{type:"input",data:"echo hi",enter:true}` (one shot WITH enter flag) **works**.

### Root cause
- **Browser** (`ddx-term-web/src/lib/term/xterm-client.ts:266 sendInput`) sends each xterm
  `onData` keystroke as `{type:"input", terminalId, data}` вҖ” **raw characters, including
  `\r` for Enter, and NO `enter` field** (xterm's onData stream is how real terminals work).
- **Gateway** (`ddx-term-broker/src/modules/gateway/term.gateway.ts:255 handleInputFrame`)
  does `send-keys -t <target> -l <data>` and only sends a proper Enter key event when the
  frame has `enter:true`. So the browser's `\r` is injected as a **literal CR via `-l`**,
  which tmux/the shell does NOT treat as "submit command." Commands accumulate but never run;
  live echo is also off.
- **Contract mismatch**: `InputFrameSchema` has an optional `enter` flag (designed for the
  agent/MCP one-shot send: `send-keys -l text` + separate Enter). The web client never uses
  it вҖ” it streams raw chars. The two halves of the input reciprocal pair disagree.

### Fix (next session вҖ” at the cause, in the gateway)
In `handleInputFrame`, when `enter` is undefined (browser path), translate control chars in
`data` into proper tmux key events: split on `\r`/`\n` вҶ’ for each segment send `send-keys -l
<segment>` then `send-keys Enter`. Keep the existing `enter:true` one-shot path for the MCP.
Alternatively (less clean): web client strips `\r` from data and sets `enter:true` вҖ” but the
gateway fix is more correct (handles arrow keys / Ctrl chars uniformly later). **Verify with
the reproduction above: per-char `ls\r` must echo the directory listing.**
After the fix, if the contract/gateway changed, rebuild broker + (if contract) the MCP bundle,
restart broker (Node 22) + web, and re-run the repro IN THE BROWSER.

### 0.C вҖ” term_destroy NOT ATOMIC: kills tmux window but leaves stale registry entry
**Status: confirmed in the 12:25 agent run. Real broker bug.**
- `term_destroy(t05)` вҶ’ first call returned `TMUX_ERROR / broker DELETE /terminals failed: 500`.
  Retry вҶ’ `can't find window: @38`. Net state: tmux window @38 GONE, but broker registry STILL
  lists `t05 вҶ’ @38` (orphan). `term_list` keeps showing the dead terminal; AC "after destroy,
  term_list no longer shows it" FAILS.
- Root cause (to confirm): broker `DELETE /terminals/:id` (`terminal.service.destroy` /
  `session.service`) kills the window THEN does registry cleanup / response serialization, and
  the post-kill step throws 500 вҶ’ registry entry never removed; the kill is not rolled back and
  isn't idempotent (second call 500s on missing window). Fix: make destroy idempotent +
  remove-from-registry FIRST or in a finally, and treat "window already gone" as success (200/204).
  File: `ddx-term-broker/src/modules/terminal/terminal.service.ts` (destroy) + session.service.

### 0.D вҖ” REGISTRY вҶ” TMUX DRIFT (both directions)
**Status: confirmed. Lower priority (mostly standalone-mode legacy).**
- The 12:25 run showed: broker registry lists `t02/t03/t05` but tmux has `t04(@30)/t06(@22)`
  the broker does NOT know, AND broker has `t05(@38)` tmux no longer has. Two sources:
  (1) earlier STANDALONE-mode MCP runs created windows the broker never registered, (2) the
  0.C destroy bug orphans entries. Fix: a reconcile/sweep on broker boot (list tmux windows,
  drop registry entries whose window is gone, optionally adopt unknown windows), and now that
  the MCP is broker-attached, no NEW standalone windows should appear. Clean current drift by
  killing stray windows or restarting the broker after `tmux kill-server` on the dev socket.

## 4. Key Decisions (WHY)
- **Node 22 for broker** вҖ” node-pty has no Node-26 prebuilt; control-mode attach needs a real pty. MCP stays Node-version-agnostic + PTY-free (invariant).
- **Single-origin WS via Next custom server** вҖ” user chose it (HTTPS-safe, broker port private). Next can't proxy WS via config rewrites (HTTP only) вҶ’ custom server owning `upgrade`.
- **Bundle contract into MCP** вҖ” `@dudoxx` org owns the MCP; `@ddx` scope isn't ours; bundling avoids publishing a 2nd package.
- **`active.default(false)`** вҖ” forgiving fix over making the broker compute tmux focus state (not needed for the demo).
