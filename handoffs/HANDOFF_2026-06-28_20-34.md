## Key Decisions
- **Live re-style, not reconnect**: separate `useEffect([appearance])` → `applyAppearance()`
  so font/theme changes mutate options in place; scrollback + WS survive. Folding into
  switchTo would reconnect on every change.
- **Themes as TS data, not CSS**: xterm consumes a hex ITheme object, so the palette lives
  in `appearance.ts` (raw-hex exemption), not globals.css `@theme`.
- **localStorage for appearance**: design-system §12/§14 — survivable-but-not-shareable state.
- **pty fix is 3-layered** (afterAll + process signals + globalSetup) because afterAll is
  skipped on interrupt — the exact path that leaked 14 servers.

## Dead Ends
- Starting the broker from the agent's Bash sandbox: impossible — no controlling TTY,
  `tmux new-session`/`openpty`/`script` all fail with `Device not configured`. User must
  launch it from a real terminal. (Web server `server.mjs` is fine — no pty.)
- Treating the `out of pty devices` error as ddx-term's fault: it isn't. The stack holds
  ~1 pty total (broker = ONE node-pty singleton; tmux = 1 server; term_create = tmux
  new-window, NOT a pty; MCP/web = 0). Real cause = other concurrent dev stacks
  (dudoxx-ai-notebook 4-app Turbo, ddx-saas nest watch) vs the system-wide 511 cap.
- `tmux kill-server` on a stale socket reports "already gone" while the process lives —
  must kill by PID. And kill signals are swallowed by the agent sandbox → use
  `dangerouslyDisableSandbox` for the reap.
- `/dev/ttys*` node count is a persistent-device-node count, not live pty usage.
