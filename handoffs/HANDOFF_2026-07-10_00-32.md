### ISSUE 0 — PTY-device LEAK via unbounded attach retry — OPEN, CRITICAL (ROOT CAUSE)
- **Symptom**: broker logs `pty.spawn(/opt/homebrew/bin/tmux) failed: posix_spawnp failed` on a
  loop; eventually macOS `kern.tty.ptmx_max` (511) is EXHAUSTED → Ghostty/any new terminal errors
  "Your system cannot allocate any more pty devices"; existing terminals can't type; broker can't
  attach.
- **Root cause**: `ddx-term-mcp` `ControlModeAttach` retries `pty.spawn(tmux -CC attach-session)`
  **every 2000ms FOREVER** on failure (see broker log "retrying in 2000ms"), with NO max-attempts
  ceiling and NO backoff cap. Each failed/partial attach leaks a PTY. Over minutes → pool
  exhaustion. (The WS client already caps at RECONNECT_MAX_ATTEMPTS=60 + backoff; the control-mode
  attach loop does NOT — it must adopt the same bound.)
- **Secondary trigger**: when `pty.spawn` itself fails (node-pty can't spawn — the ORIGINAL reason
  attach failed, before exhaustion), the loop spins hot. Investigate WHY `pty.spawn(tmux)` fails
  when plain `execFile(tmux)` succeeds — likely node-pty `spawn-helper`/`pty.node` deploy issue in
  `dist/broker/node_modules` (NOTE: `spawn-helper`/`pty.node` were NOT FOUND under dist/broker —
  the `pnpm deploy --legacy --prod` in build-stack.mjs may be dropping node-pty's native prebuilt).
- **Recovery (done this session)**: `pkill -f dist/broker/main.js` (stop the leak loop), sweep all
  `tmux -S /tmp/*` servers, clear sockets. Verified PTY fork works again (was 511-exhausted → 63).
- **THE FIX**: (a) cap the control-mode attach retry (max attempts + exponential backoff, mirror
  the WS reconnect); (b) on give-up, surface a clear error instead of looping; (c) fix the
  underlying node-pty spawn failure (verify spawn-helper is deployed + executable + arm64 in
  dist/broker/node_modules). Without (a), any attach failure WILL exhaust host PTYs again.
- **Broker is currently STOPPED** — do NOT restart `dist/broker/main.js` until the retry loop is
  bounded, or it re-leaks PTYs. Web :13340 is up but broker-less.

### ISSUE 2 — MCP `term_create` reclaim-throw race — OPEN
- **Symptom**: `mcp__ddx-term__term_create` (and the whole stack launch) throws
  `{"code":"STACK_LAUNCH_FAILED","message":"broker lock was stale and reclaimed — retry"}` on the
  first call, EVEN WHEN the broker is already healthy on :13330.
- **Root cause**: `ddx-term-mcp/dist/server.js` `pollUntilHealthy()` (line ~731-748): each poll
  iteration, if health check hasn't passed yet, calls `reclaimStaleLock(BROKER_LOCK_PATH)` which
  returns `true` when the lock file is absent (line 614-615) → **throws immediately** instead of
  giving the detached broker its 30s `BROKER_READY_TIMEOUT_MS` window. So a cleared/absent lock
  makes the very first poll throw. Source: `src/supervisor/ensure-stack.ts` (bundled into
  server.js). Fix: only throw on reclaim if the lock existed AND was held by a live-but-stale
  holder; an ABSENT lock during our own spawn is normal — keep polling until timeout.
