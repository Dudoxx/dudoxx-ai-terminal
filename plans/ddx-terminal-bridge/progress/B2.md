## Deviations
- DEV-1 (real bug found+fixed via e2e): **constant-height capture trap**. `capture-pane -p` pads to the
  full viewport with trailing blank lines, so naive line-count never grows as output scrolls and the
  delta cursor never advanced (e2e "delta read" failed). Fix: `capture-util.stripTrailingBlank()` strips
  ALL trailing blanks before the cursor math, in both term_read and term_snapshot.
- DEV-2 (real bug found+fixed): **entry-point guard**. `import.meta.url === file://${argv[1]}` never matched
  (argv[1] may be relative/unencoded and import.meta.url is symlink-resolved). Fixed to
  `import.meta.url === pathToFileURL(realpathSync(argv[1])).href`. Guard proven true for an absolute path under node.
- DEV-3 (advisory, intentional): resolver-factory uses raw `fetch()` (JIT-steer flagged retry/timeout). The
  broker call is same-host localhost dev; `FetchLike` is injectable so a hardened impl can be supplied by the
  broker integration. Not over-engineering a thin same-host resolver. Marked here, not auto-changed.

## BLOCKER (escalated to main) — cross-package, OUT OF MY BOUNDARY
The compiled `dist/server.js` (the `bin` / `.mcp.json` launch target, ESM) CANNOT start under plain `node`:
```
ERR_MODULE_NOT_FOUND: Cannot find module '.../packages/ddx-term-contract/dist/esm/terminal'
  imported from .../packages/ddx-term-contract/dist/esm/index.js
```
Root cause: `@ddx/term-contract`'s ESM build emits EXTENSIONLESS re-exports (`export * from './terminal'`).
Node's ESM resolver requires explicit `.js` extensions; the contract's `tsconfig.esm.json` uses
`moduleResolution: Bundler` which does NOT rewrite specifiers. The CJS build resolves fine
(`require('@ddx/term-contract')` → 50 keys), and test runners (vitest/tsx) resolve extensionless imports,
which is why all 34 tests pass — but the production ESM `bin` does not load.

Fix belongs in `packages/ddx-term-contract/` (A2's package — I must NOT edit it). Smallest fix there, ONE of:
1. add `"type": "module"` + emit `.js`-extensioned specifiers (set `module: NodeNext`/`moduleResolution: NodeNext` in tsconfig.esm.json), OR
2. add a `tsc-alias`/post-build step that appends `.js` to the ESM relative specifiers, OR
3. point the package `exports.import` away from the broken build until rebuilt.

My package is complete and correct; it will launch the moment the contract ships a Node-resolvable ESM entry.
