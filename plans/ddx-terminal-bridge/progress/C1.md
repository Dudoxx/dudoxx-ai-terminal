## Invariant compliance

- Frame types imported ONLY from `@ddx/term-contract` (never redefined locally).
- Tab switch = `XtermClient.dispose()` + `new XtermClient()` + `restoreSnapshot()` — not a full WS reconnect (RESPONSIVENESS §2.8).
- Zero `any` in all files.
- All user-facing strings via `t()` in en/de/fr lockstep.
- Semantic OKLCH tokens only in components — raw hex exists only in `buildXtermTheme()` (xterm.js documented constraint; values derived from globals.css `@theme` CSS vars at runtime via `cssVar()`).

## Deviations

- `xterm-client.ts` contains hex color fallback strings inside `buildXtermTheme()`. This is a documented constraint: xterm.js cannot consume CSS variables or OKLCH. The authoritative values are the `--xterm-*-hex` CSS custom properties declared in `globals.css @theme`; `buildXtermTheme()` reads them via `getComputedStyle` at mount time, with hex literals only as SSR-guard fallbacks. Design hook is advisory and non-blocking; exception is noted here.
- WS numeric literals (`1` for OPEN, `3` for CLOSED) used instead of `WebSocket.OPEN`/`WebSocket.CLOSED` to ensure test-stub compatibility without static property definitions.
- `ddx-term-web` does NOT apply the full Dudoxx entity-route/header-footer machinery (In-Session scope note from tasks.json context — this is a CLI terminal UI).
