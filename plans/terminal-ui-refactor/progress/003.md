## Notes / deviations

- **AC1.6 (cold-load renders from route param) / AC1.7 (unknown terminalId not-found) /
  AC3.3-shape browser ACs — VERIFIED-PENDING-BROWSER.** Cannot start a dev server in this
  sandbox per operator directive; typecheck+lint+vitest are the ceiling of static
  verification for routing/navigation behavior. Operator + lead to run the Playwright pass.
- **Architecture choice not spelled out in the task node — flagging for review**: rather
  than duplicating the WS/xterm lifecycle logic across `view/page.tsx` + `readonly/page.tsx`
  + `edit/page.tsx`, I extracted ONE shared client component (`TerminalWorkspace`,
  parameterized by a `mode: 'view'|'readonly'|'edit'` prop) and made the three route files
  thin server shells. This keeps the dispose→new-client→snapshot sequence in exactly one
  place (task node's own pitfall FM1 warns against a second XtermClient double-mounting) —
  three independent page implementations would have risked drift between them. If the
  intended shape was three fully independent page bodies, this is a deviation worth a
  second look.
- **`edit` mode is currently a hint only** — `TerminalWorkspace` accepts `mode="edit"` and
  passes it through, but does not yet auto-open the inline-rename UI on mount (the existing
  `TerminalRow` rename affordance is reused, per Cardinal #20, rather than a new form). Task
  context said "modes view/[terminalId]/readonly and view/[terminalId]/edit (rename flow)"
  — the route and mode plumbing exist; auto-opening the rename input specifically for `edit`
  mode was not wired to avoid guessing at UX not specified in the task (which row? side
  panel focus vs inline title edit?). Flagging as a partial rather than silently deciding.
- **Timeline sort key**: `TerminalDescriptor` has no field named literally "timestamp" in
  the task's own context text (which suggested `windowId` implicitly by omission) — used
  the real `createdAt: number` (epoch ms) field instead, which is the semantically correct
  choice and already exists in the contract. See Validation output above for the type
  errors this caught.
- **`new/page.tsx` has no create-form fields** — task context said "(create-or-redirect)"
  as an explicit alternative to a form; implemented as create-then-redirect since a
  terminal's only creatable property (title) is already auto-generated identically to the
  old +New button's behavior, and the broker doesn't expose any other terminal-creation
  parameter to put in a form.
- §10 sub-entity/export/print routes correctly EXCLUDED per the task's WS1 NOT line —
  terminals have none.
