## Steps done
- [x] Read web-audit.md + xterm-client.ts to map connect()/onmessage/restoreSnapshot and
      the `this.disposed` guard.
- [x] Added a frame buffer: `private painted = false` + `private pendingFrames:
      ServerFrame[] = []`. Reset in `openWebSocket()` on EVERY attach AND reconnect (not
      just the first connect) — a dropped/resumed socket can race live frames against the
      reconnect repaint the same way a cold attach races the initial snapshot.
- [x] `handleServerFrame()` now: (a) intercepts a `snapshot` frame and routes it directly
      through `restoreSnapshot()` (consuming the broker's authoritative cold-attach frame
      from task_002/task_001 without a separate REST call); (b) buffers every other frame
      FIFO while `!painted`; (c) writes immediately once painted.
- [x] `restoreSnapshot()` (the REST-fallback / onReconnect path) now calls a new
      `markPaintedAndFlush()` after writing, so BOTH convergence paths (WS `snapshot` frame
      and REST fallback) share one flip-and-flush contract — whichever arrives first wins
      the race and buffered frames flush in original arrival order.
- [x] Extended the disposed guard to the new flush path: `restoreSnapshot()` returns
      immediately if `this.disposed`; `markPaintedAndFlush()` re-checks `this.disposed`
      too (defense in depth, since it's the one place that mutates `painted`). `dispose()`
      also clears `pendingFrames` (no dangling refs past teardown).
- [x] Extracted the per-frame-type switch into a new private `writeFrame()` method (the old
      switch body, unchanged logic) so `handleServerFrame()` (buffer-or-write decision) and
      `markPaintedAndFlush()` (buffer replay) share one write path — no duplicated switch.
- [x] Checked `view/[terminalId]/page.tsx` (named in the task's `files.modify`, written
      before task_003 existed — see Deviations): the actual mount/switch ordering now lives
      in `TerminalWorkspace.tsx` (task_003's extraction), which already does
      `connect()` → `fetchSnapshot()` → `restoreSnapshot()` in that exact order. No change
      needed there — the buffering guarantee is now fully internal to `XtermClient`, so
      the call site didn't need to change shape.
- [x] Added 3 new `xterm-client.spec.ts` cases: (a) live frames before the snapshot are
      buffered and flush in FIFO order after `restoreSnapshot()`; (b) a broker `snapshot`
      frame paints before a buffered live frame (consumed via `handleServerFrame`, not a
      manual `restoreSnapshot()` call); (c) `restoreSnapshot()` after `dispose()` performs
      no write (stale-client guard). Also updated the pre-existing AC#7 test, which wrote a
      bare live frame with no prior paint — now correctly buffered under the new contract,
      so the test calls `restoreSnapshot('')` first to match the real page's connect() →
      restoreSnapshot() flow.
- [x] Ran web typecheck/lint/test.

  route family existed) — this is the documented "plan-write staleness" pattern (task

  003's extraction of the old page.tsx body) already calls `connect()` then
  `restoreSnapshot()` in the correct order, so no call-site change was required. Flagging

- **Did NOT rewrite reconnect-backoff** (WS3 NOT line) — the jittered exponential
  backoff in `scheduleReconnect()` is untouched; the only change to `openWebSocket()` is
  the two-line `painted`/`pendingFrames` reset at the top.
- **Did NOT change the `@ddx/term-contract` schema** — `SnapshotFrameSchema` (task_001) is
  consumed as-is; no contract edits.
- **No double-decode**: `frame.data` on a `snapshot` frame is passed straight to
  `restoreSnapshot()`, which only does its existing bare-LF→CRLF normalize (documented as
  already necessary for the REST path's `tmux capture-pane -p` output) — no additional
  `\033`/`\015` decode was added, since the broker (task_002) already decodes those to raw
  bytes before the frame reaches the wire.

  exhaustiveness — that frame type is intercepted earlier in `handleServerFrame()` and
  never actually reaches `writeFrame()` (documented inline in the code).
